# جسور جلوبال — منصة البرامج السياحية
### Jusoor Global — Tourism Programs Platform

منصة متكاملة لإدارة البرامج السياحية: قاعدة وجهات هرمية، إدارة كاملة للمحتوى،
رفع صور، بناء برنامج يومي تلقائي، وتصدير برنامج احترافي للعميل — **ثنائية اللغة عربي/إنجليزي**
مع تبديل فوري للاتجاه (RTL/LTR).

> منصة كاملة مبنية على React + Vite + Supabase، متوافقة مع هوية جسور (كحلي #171d48 · ذهبي #be9a30 · خط Cairo).

## ✦ ثنائية اللغة

- زر تبديل اللغة في الأعلى (ع / EN) يبدّل **كل الواجهة والاتجاه فورًا**.
- كل بيانات الوجهات تُحفظ بالاسمين (`name_ar` / `name_en`, `title_ar` / `title_en`)، ويعرض النظام المناسب للغة.
- المستند الاحترافي يتبع لغة الواجهة (RTL للعربية، LTR للإنجليزية).

## ✦ الإعداد (مرة واحدة · ~١٠ دقائق)

### ١. أنشئ مشروع Supabase
[supabase.com](https://supabase.com) ← *New project* (الخطة المجانية تكفي).

### ٢. نفّذ المخطط
*SQL Editor ← New query* → الصق كامل `db/schema_v2.sql` → *Run*.
(يُنشئ كل الجداول، العلاقات، سياسات الأمان RLS، وbucket الصور `media`.)
ثم نفّذ `db/migration_03_customers_invoices.sql` بنفس الطريقة (يضيف العملاء والفواتير والدفعات).
ثم نفّذ `db/migration_04_reports_email.sql` (يضيف التقارير وصندوق البريد والإعدادات).
ثم نفّذ `db/migration_05_company.sql` (يضيف بيانات الشركة للمستندات).
ثم نفّذ بالترتيب: `migration_06_flights.sql` (الطيران المستقل)، `migration_07_notes.sql` (ملاحظات البنود)،
`migration_08_fixes.sql` (إصلاحات البرنامج اليومي)، `migration_09_suppliers.sql` (المزودون)،
`migration_10_itinerary_bilingual.sql` (ثنائية اللغة للبرنامج)، `migration_11_marketing_quotes.sql` (العروض التسويقية).

> **إشعارات البريد (اختياري):** بعد الإعداد، انشر وظيفة الإرسال:
> `supabase functions deploy send-emails` ثم `supabase secrets set RESEND_API_KEY=...`
> وجدوِلها كل بضع دقائق من Edge Functions → Schedules. بدونها تبقى الرسائل في «الانتظار».

### ٣. أنشئ أول مدير
- *Authentication ← Users ← Add user* (مثلاً `admin@jusoor.global` + كلمة مرور).
- انسخ الـ **User UID**.
- في *SQL Editor*: الصق `db/seed_admin.sql` بعد وضع الـ UID مكان `<AUTH_UID>` → *Run*.

### ٤. اربط المفاتيح
انسخ `.env.example` إلى `.env` واملأ من *Project Settings ← API*:
```
VITE_SUPABASE_URL=https://xxxxx.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbGc...
```

### ٥. شغّل
```bash
npm install
npm run dev        # تطوير محلي
npm run build      # نسخة الإنتاج (مجلد dist)
```

## ✦ النشر على الإنترنت

**Vercel (موصى به):** ارفع المشروع على GitHub ← [vercel.com](https://vercel.com) *Import* ← أضف
متغيّري البيئة (`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`) في *Settings → Environment Variables* ← Deploy.
كل دفعة لاحقة تُنشر تلقائيًا.

**Netlify:** نفس الفكرة — اربط المستودع، أضف المتغيّرين، Build command: `npm run build`, Publish: `dist`.

> لا تنشر مجلد `dist` يدويًا دون متغيّرات البيئة — التطبيق يحتاج مفاتيح Supabase ليعمل.

## ✦ كيف تستخدمها (تدفّق العمل)

1. **ابنِ قاعدة الوجهات** من *الوجهات*: أضف دولة ← مدنها ← فنادقها/جولاتها/أنشطتها، وارفع الصور.
2. **عرّف المواصلات** من *المواصلات*: خطوط النقل بين المواقع (مطار↔فندق، مدينة↔مدينة).
3. **أنشئ طلبًا** من *الطلبات*: أدخل بيانات العميل، ثم أضف البنود من الكتالوج (الأسعار تُملأ تلقائيًا وتُعدّل).
4. **ولّد البرنامج اليومي**: زر *البرنامج اليومي* ← *توليد من التواريخ* ← عدّل الأيام والعناصر ← احفظ.
5. **صدّر البرنامج**: زر *عرض البرنامج الاحترافي* ← *طباعة / PDF* ← شاركه مع العميل.

## ✦ ملاحظات معمارية

- **هرمية صارمة:** كل فندق/جولة/نشاط مربوط بمدينة عبر مفتاح خارجي، وكل مدينة بدولة. اختيار الدولة يُظهر مدنها فقط.
- **المواصلات مستقلة:** كتالوج خطوط نقل قابلة لإعادة الاستخدام بين أي موقعين.
- **الصور في التخزين لا في القاعدة:** تُرفع لـ Supabase Storage وتُربط روابطها — أداء أفضل.
- **أمان حقيقي:** RLS على كل جدول حسب صلاحيات المستخدم (إضافة/تعديل/حذف).
- **الإجماليات محسوبة لحظيًا** عبر view، وعمود الربح مولّد تلقائيًا — لا تخزين مزدوج.
- **ترقيم الفاتورة آلي** عبر trigger عند تحويل الطلب إلى «منجز».
